2,000-Step LLM Speedrun

Train a small GPT from scratch on ~7 MB of mixed English + Hindi text, CPU only, under hard caps: ≤ 2,000 optimizer steps, ≤ 2,000,000 parameters, provided corpus only, pure PyTorch, no pretrained weights. Metric: bits per UTF-8 byte (bpb) on a hidden test file — lower is better.

Central finding: the binding resource was optimizer updates, not parameter capacity. Every change that helped increased useful learning per update; every change that hurt either added capacity that could not be trained in 2,000 steps or reduced gradient throughput. The final 1.728-bpb model is deliberately not the largest permitted model — it is the one that learns most effectively within 2,000 updates.

Constraint compliance

  • ≤ 2,000 optimizer steps — checkpoint records exactly 2,000
  • ≤ 2,000,000 parameters — 1,421,760
  • CPU only, pure PyTorch + numpy + stdlib, no pretrained weights
  • Tokenizer trained only on train_corpus.txt; lossless on arbitrary UTF-8 (byte fallback)
  • Official evaluate.py command works unchanged (byte-for-byte identical to the handout)

Pipeline

UTF-8 text→ byte-level BPE (vocab 1,024)→ GPT (4 layers, d=160, tied embeddings)→ next-token probabilities→ cross-entropy ÷ bytes = bpb

Final architecture and training configuration

Architecture (~1,421,760 params)

  • Byte-level BPE, vocab 1,024 (256 bytes + 768 merges), ~2.9 bytes/token
  • 4 Transformer layers, 4 attention heads
  • Embedding width 160, context length 128 tokens
  • Tied input/output embeddings
  • LayerNorm + GELU MLP (4x), causal attention
  • No dropout; N(0, 0.05) initialization

Optimization

  • AdamW, betas (0.9, 0.95)
  • Peak LR 1.4e-3 (re-tuned to the larger batch), 100-step warmup, cosine decay to 0.1x
  • Weight decay 0.1 on matrix/embedding params only (none on biases/norms)
  • Gradient clipping 1.0
  • Batch size 64, block 128, exactly 2,000 steps

How the model works (in plain terms)

  • Token embeddings turn each BPE token id into a learned vector.
  • Position embeddings add order information so the model knows where each token sits.
  • Causal self-attention lets each position gather information from earlier positions only, never future ones.
  • MLP (feed-forward) transforms the information each position collected via attention.
  • Residual connections add each block's output back to its input, preserving signal and stabilizing optimization.
  • Tied output layer reuses the token-embedding matrix to turn the final vectors into next-token scores.
  • Cross-entropy trains the model to minimize the negative log-probability of the true next token.

The tokenizer, and why bpb stays fair

  • Starts from all 256 possible bytes, so any byte sequence is always representable.
  • Repeatedly merges the most frequent adjacent pair into a new token — 768 merges learned from train_corpus.txt only.
  • Raw bytes remain as a fallback, so arbitrary UTF-8 (unseen English or Hindi) round-trips losslessly: decode(encode(text)) == text, which the scorer verifies.
  • The score divides total loss by the number of original UTF-8 bytes, not tokens, so tokenizer choices stay comparable across candidates. This is why a "worse-looking" per-token loss can still be a better bpb.

Three quantitative reasons the winners won

  1. Tokenizer buys context. The byte tokenizer is ~1 byte/token; BPE is ~2.9 bytes/token, so the same 128-token context now spans ~370 bytes instead of 128, and Hindi characters stop costing 3 tokens each.
  2. Batch size exploits the step cap legally. Steps are capped, tokens are not: over 2,000 steps, batch 8 sees ~2.05M target tokens, batch 32 ~8.19M, batch 64 ~16.38M. More data per update, same number of updates.
  3. The full parameter allowance is not used on purpose. Run 7 spent nearly the whole 2M budget and got worse bpb at 2.8x the wall-clock, because that capacity cannot be trained in 2,000 steps.

What worked, what failed, and why

What worked

  • Optimizer + schedule (Run 2): the baseline was simply under-optimized within the budget.
  • Bigger batch (Runs 5, 8): the dominant lever — more useful gradient signal per fixed step.
  • LR re-tuned to the batch (Run 10): raising peak LR to 1.4e-3 to match the 8x-larger batch gave the final drop to 1.728.
  • Corpus-trained BPE (Run 6): more effective context per prediction and fewer wasted tokens on Devanagari.
  • Weight tying: kept for parameter efficiency at roughly neutral score.

What failed (and the lesson)

  • GPT-2 init (Run 4): small init learns too slowly — the run is too short for it to pay off.
  • Larger 1.9M model (Run 7): capacity was undertrained; maximum allowed size is not automatically optimal.
  • Suffix loss (Run 9): aligning training to the scorer's context window halved supervised targets per update, so throughput fell.

The three failures share one cause and reinforce the central finding: at a 2,000-step budget, anything that reduces useful gradient signal per step — whether through slow-starting init, untrainable capacity, or fewer targets — loses, even when the idea is individually reasonable.

What I would try next (with more step or wall-clock budget)

  • Batch 96–128 with a re-tuned peak LR (~1.25e-3, shorter warmup) — batch was still improving at 64; stopped only for CPU wall-clock.
  • BPE vocab sweep (768 / 1,280 / 1,536) — 1,024 was a first guess; there is an unsearched sweet spot between compression and embedding cost.
  • RMSNorm, bias-free linears, SwiGLU MLP — modern components tested one variable at a time.
  • Coverage-balanced sampling — shuffled full-corpus coverage instead of random-with-replacement window starts.

Reproducibility

All commands run inside this folder.

# (only if bpe_merges.json is missing) train the tokenizer on the corpus
python train_bpe.py --data ../data/train_corpus.txt --vocab_size 1024

# train the final model (2,000 steps, batch 64, AdamW + warmup/cosine)
python train.py --data ../data/train_corpus.txt --steps 2000 --lr 1.4e-3 --batch 64 --out ckpt.pt

# official evaluation (unchanged)
python evaluate.py --checkpoint ckpt.pt --text_file ../data/dev_eval.txt
# -> {"bpb": 1.728, "n_params": 1421760, "steps": 2000, ...}

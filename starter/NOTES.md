# NOTES

The final model is a 4-layer, 4-head GPT with embedding width 160, context 128 tokens, tied input/output embeddings and no dropout, totalling about 1,421,760 parameters (well under the 2,000,000 cap). It is trained for exactly 2,000 steps with AdamW (betas 0.9/0.95, peak LR 1.4e-3, 100-step warmup, cosine decay to 0.1x, weight decay 0.1 on matrix/embedding parameters only, gradient clipping 1.0) at batch size 64, reaching 1.728 dev bits-per-byte versus the 2.3718 baseline (about 27% lower). The binding resource was optimizer updates, not parameter capacity, so every gain came from increasing useful learning per update rather than from a larger model. A corpus-trained lossless byte-level BPE (256 raw bytes + 768 merges, vocab 1,024) compresses roughly 2.9 bytes per token, so the same 128-token context covers about 370 bytes instead of 128 and Hindi text stops costing three tokens per character. Larger batches were the strongest lever because they feed more target tokens into each of the fixed 2,000 steps (batch 8 -> 32 -> 64 moved bpb 2.24 -> 1.96 -> 1.76), and re-tuning the peak LR upward to 1.4e-3 to match the 8x-larger batch gave the final step to 1.728. The proper optimizer and schedule alone recovered 2.37 -> 2.24, confirming the baseline was simply under-optimized within the step budget. A deliberately larger 1.9M-parameter model (vocab 2,048, context 256, five layers) scored slightly worse (1.835) at 2.8x the wall-clock, because that capacity cannot be trained enough in 2,000 steps. For the same reason the final model intentionally stays well below the 2M cap: extra parameters were untrainable overhead, not free accuracy. Weight tying is kept because it frees parameters at neutral cost, and plain N(0, 0.05) initialization beat the textbook GPT-2 scaled init because fast early learning matters more than depth-stable signal propagation over so few steps.

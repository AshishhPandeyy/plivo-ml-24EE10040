# NOTES

Best config: byte-level **BPE tokenizer (vocab 1024)** trained only on the corpus, feeding the original 4-layer / 4-head / d=160 GPT (block 128, tied weights, 1.42M params), trained for the full 2,000 steps with **AdamW (betas 0.9/0.95, weight decay 0.1 on 2D+ weights), 100-step warmup + cosine decay to 0.1x, peak LR 1e-3, grad-clip 1.0, batch 64**. Dev score: **1.76 bpb** (baseline was 2.3718).

Why it works: the binding constraint is the 2,000-step cap, not parameters, so every gain came from learning more *per step* rather than from a bigger model. The optimizer/schedule fix (constant Adam 3e-4 -> warmup+cosine AdamW 1e-3) alone took 2.37 -> 2.24. BPE compresses ~2.9 bytes/token, so the same 128-token block covers ~2.9x more real context and the Devanagari-costs-3-bytes problem disappears (2.24 -> 1.83). Enlarging the batch (8 -> 32 -> 64) was the other big lever because it feeds more tokens into each of the fixed 2,000 steps (down to 1.76). I deliberately tested "just max the parameter budget" (1.9M params, block 256, 5 layers) and it did *not* help — the bigger model was undertrained at 2,000 steps and scored slightly worse (1.835) at 2.8x the wall-clock, which is the clearest evidence that step-count, not capacity, is the bottleneck. Weight tying is kept (frees params, neutral on score) and the plain N(0,0.05) init beat the textbook GPT-2 scaled init because large early logits/gradients matter more than depth-stable signal propagation when you only have 2,000 steps.
